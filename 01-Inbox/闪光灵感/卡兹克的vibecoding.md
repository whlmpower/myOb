# 网页标题


这篇文章是**数字生命卡兹克**写的 Vibe Coding 工作流实践总结。以下是完整提炼：

---

### 一、前期核心投入：测试流程优化

> "前期我把巨量的 Token 都消耗在了一个地方，那就是优化我的整体测试流程上。"

他的核心判断：**Vibe Coding 中写代码已不再是瓶颈，瓶颈转移到了测试、验证和对方案的评审上。** 所以前期最重要的投入不是编码，而是搭建一套可靠的测试流水线。

---

### 二、自建 CI 服务器 + 任务路由

- 没有使用 GitHub 托管的 Runner
- **单独在腾讯云上租了一台便宜的服务器做 CI**
- 做了**多种任务类型的路由**（task-type routing）
- 目的：在保证测试全面性和准确性的前提下，**节省整体的测试和部署时间**

---

### 三、完整自动化交付闭环流程

```
发任务后
→ Agent 理解并核实问题
→ 新建独立分支和工作区
→ 开发修改
→ 自动测试，必要时回测或检查页面
→ 推送分支并创建 PR
→ CI 自动验收
→ 合并到主分支
→ 部署上线
→ 检查真实线上效果
→ 洁癖.skill 同步代码、文档和 Agent 记忆，并复盘经验
→ 向作者汇报最终结果
→ 作者确认没有问题
→ 清理开发分支、工作区和临时数据库
→ 任务结束
```

关键数据：

- Agent 提测 → 测试约 **5 分钟** → 被打回 → 修复 → 再提测 → 又是 5 分钟
- 合并前有 CI 自动验收关卡，**过不了就无法合并**
- 上线后还有**真实线上效果检查**
- 最后通过**洁癖.skill**做代码、文档、规则、记忆**四端统一**并复盘经验

---

### 四、对测试流程的自信来源

- **测试覆盖率做到了比较舒服的区间**
- 因为测试流程足够严格，所以对 Codex 的"目标模式"有足够信心——"如果有一些 BUG 大概率都是过不了会被打回来重新优化的"
- 可以放心让 Agent **长时间跑**（最长跑过 17 小时），因为测试关卡会把问题拦住

---

### 五、核心思路：哑铃型架构

```
        ┌──────────────────────────────────────┐
        │          哑  铃  形  态                │
        └──────────────────────────────────────┘

   左端（方案与执行）                    右端（质量与统一）
   ──────────────────                   ──────────────────
   用最牛逼的模型出方案                  最重要的测试和研究流程
   Claude Fable 5 → GPT-5.6 Sol         洁癖.skill 保证代码/文档/
   → Codex 目标模式全自动执行            规则/记忆 四端统一
```

---

### 六、实践细节

|要点|说明|
|---|---|
|Codex 目标模式 > Claude Code|GPT 系列幻觉率低、Prompt 遵从好，超长程任务几乎不变形；Claude 有时会陷入死循环|
|6-7 个任务并行是极限|**作者自己（人脑确认）才是整个流程的最大瓶颈**|
|传统开发 skill 已无用|如 superpowers 这类 skill，不如这套流程有效|
|不要用 Fable 5 开目标模式|太贵，不值当|
|小任务不用目标模式|直接让 GPT-5.6 Sol 按流程执行完毕即可|

---

### 七、一句话总结

> **前期花大量 Token 搭建严格的自动化测试 CI 流水线 + 全自动闭环流程（开发→测试→CI验收→合并→部署→人工确认→清理），让测试关卡成为质量的最后防线，从而可以放心地把长时间的任务交给 Agent 自主执行。**
